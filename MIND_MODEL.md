# MIND_MODEL — 作業姿勢の心的モデル

このドキュメントは、特定プロジェクトの仕様ではなく、**どんなプロジェクトでも通用する一般的な進め方・考え方**をまとめたものです。
コード上の細かな命名やC++スタイルは `RULE_EMBEDED_CPP_SIMPLE.md` / `RULE_EMBEDED_CPP.md` を参照してください。本書はその手前の「どう考え、どう進めるか」を扱います。

各項目は **原則 / なぜ / どう適用するか** の3点で記述します。

---

## 0. このユーザーについて

- 組込みオーディオ開発者。設計トレードオフを慎重に考え、コミット前に良い問いを立てる。
- 実装に入る前に設計ドキュメントをレビュー・検証したい。**設計意思決定の主権はユーザーが持つ。**
- 日本語でやり取りする。

---

## 1. 設計哲学：シンプルさを最優先する

**原則**
- 防御的・複雑な解より、まずシンプルな解を提示する。
- 「念のため」のクリア/初期化/コピーなど、余計な処理を足さない。
  - 例: クリア操作はバッファを実際に消さず、状態だけ変える（次回上書きで自然に消える）。これにより undo も自明になる。
- リソースは可能なら起動時に静的確保する。動的確保は柔軟だが、確保失敗・断片化・予測不能性を持ち込む。
- 性能は**先回りして最適化しない**。実測でボトルネックが出てから対処する。

**なぜ**
ユーザーは運用上のシンプルさと、無駄な処理がないことを重視する。複雑さは将来のバグ・不整合の温床。「ユーザーが操作を間違えたら？」を常に考えており、状態復元による undo を好む。

**どう適用するか**
解を提案するときは最もシンプルな案を先頭に置く。複雑さを足すときは「なぜそれが必要か」を明示する。どんな小機能でも **「undo は必要か」「将来拡張との整合性は」** を検討項目に入れる。

## 2. 開発ワークフロー：実装前に設計を固める

**原則** — 非自明な機能は、いきなり実装せず次の順で進める。
1. **調査 → 懸念ドキュメント**：対応方針と懸念事項を番号付きドキュメント（例: `docs/NN_xxx_concerns.md`）に列挙する。
2. **1項目ずつ対話で方針確定**：曖昧な判断を残したまま実装に入らない。決めるのはユーザー。
3. **「他に懸念は？」の確認フェーズ**を必ず挟む。
4. **実装 → ビルド確認**：実装中に判明した追加懸念はドキュメントに追記する（後日の仕様根拠になる）。
5. **issue 単位で 1 コミット**：複数 issue をまとめた bundled commit は避ける。

**なぜ**
ユーザーが設計の主権を持ちたい。トレードオフを実装前に可視化してレビューしたい。ドキュメントが後日の「なぜこうしたか」の根拠になる。

**どう適用するか**
- 横断的機能（エラー表示、インジケータ、UI通知など）は元 issue に混ぜず、**別 issue に分離**する。
- I/O などの長時間処理は、バッチ一括ではなく**ステートマシンによる非同期逐次処理**（1反復＝1操作単位）を優先する。

## 3. デバッグ方法論：再現できないなら、まずテスト環境を疑う

**原則**
ハードなタイミング/非同期バグは、デバイス計装の往復を繰り返さない。
**「単体テストで再現できない＝テスト環境が実機を再現できていない」** と疑い、テストハーネス／モックを実機に寄せて**決定論的に再現**してから直す。

**なぜ**
推測ベースのデバイス往復は堂々巡りになりやすく、ユーザーはそれを強く嫌う。再現できないこと自体が「テスト環境の穴」を指し示す手掛かり。
（実例: モックのクロックがコールバックを発火させておらず、全テストで特定の同期ロジックが一度も走っていなかった、という穴が見つかったことがある。）

**どう適用するか**
再現しないバグはまずモックの忠実性を疑う。必要なら実コンポーネントを疑似タイマー等で駆動し、実機の挙動を取り込む。直すのは「現象」ではなく「再現できる土台を作ってから」。

---

## 4. 横断的ロジックは1箇所に集約し、全経路を揃える

**原則**
- モードや画面をまたぐ共通操作（修飾キーの組合せ等）は、各派生先に散らさず**共通の基底レイヤに集約**する。固有処理だけを派生側に置く。
- 入力をブロックする窓を作るときは、目立つ経路（ボタン等）だけ塞いで、別経路（外部トリガ、MIDI、イベントバス等）を素通しにしない。**全入力経路を洗い出し、同タイミング・同挙動で揃える。**

**なぜ**
共通操作を派生先ごとに実装すると、追加時に漏れ・不整合が生じる（同じ操作がモードごとに別挙動になる）。
入力経路が非対称だと、塞いだはずの操作が裏口から発火し、データ不整合を起こす。ユーザーはこのクラスの不整合を特に嫌う。

**どう適用するか**
入力ブロックや横断操作を追加・レビューするときは、**全経路を全数調査**して同じ扱いにする。集約できない複合操作が出たら、握りつぶさず都度相談して例外判断する。

## 5. コードの一貫性：既存の流儀と標準イディオムに従う

**原則**
- プロジェクト全体の命名・スタイルに合わせる。1ファイル内の定数・テストコードも例外なく統一する。Google風の `kFooBar` のような独自接頭辞を勝手に持ち込まない。
- 手書きの条件分岐より**標準ライブラリのイディオム**を使う（例: 範囲クリップは `if` の連鎖ではなく `std::clamp(v, lo, hi)`）。意図が読み手に伝わるコードを優先する。

**なぜ**
浮いた命名・流儀はレビューで必ず指摘される。標準イディオムは意図が一目で伝わり、バグも減る。

**どう適用するか**
初見の領域に手を入れる前に、その領域の既存コードと `coding_rules/` を確認してから書く。

## 6. ドメイン原則：オーディオ／音楽タイミング

> このプロジェクト固有ではなく、オーディオ機器全般に通用する設計スタンス。

**原則**
- **聴覚の許容は厳しい**：位相/遅延のズレは知覚される（訓練された音楽家は ~5ms を知覚する前提で設計）。ms単位の誤差を安易に「実用上同等」「知覚不能」と主張しない。
- **視覚の許容は緩い**：残光現象で時間分解能が低く、~100ms 程度の待機は UI で見せても知覚されない。聴覚と視覚で許容基準を使い分ける。
- **マスター/チェイン運用を常に考慮**：自機がクロックマスターになる運用では、位相が恒久にズレる解はスレーブ運用で無害でも却下される。
- **外部仕様の曖昧さは設定で吸収**：MIDIクロックのように機器依存で曖昧な仕様は、全機器対応を狙わない。安易な自動判別をせず、必要ならシステム設定で切り替える。

**どう適用するか**
タイミング/同期系の提案では、位相誤差の知覚可否を根拠に近似を正当化しない。出力クロック（CLK OUT / MIDI TX）への影響を必ず検討項目に入れる。

## 7. ツールの好み

**PDF → Markdown 変換**
- **複雑な PDF**（データシート、表・多段組）→ **Docling**（表をMarkdown表として高精度に再構成。画像は `--image-export-mode placeholder` で肥大化回避）。
- **単純なテキスト主体の PDF** → **markitdown**（高速。ただし表は崩れる）。
- **Marker (marker-pdf) は使わない／再提案しない**（Apple Silicon で非実用的に遅いと判定済み）。
